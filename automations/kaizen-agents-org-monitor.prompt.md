Monitor the Kaizen Agents work across the local repositories and GitHub remotes as a cross-repo coordinator. Active GitHub repositories: kaizen-agents-org/.github, kaizen-agents-org/builder-agent, kaizen-agents-org/kaizen-loop, and kaizen-agents-org/verifier. `coderabbit` and `renovate-config` are not active monitor targets; mention them only as downstream sync targets when `.github` sync evidence requires it.

This automation is the maintain layer defined by the organization automation roles doc. In a checkout of `kaizen-agents-org/.github`, read it at `docs/automation-roles.md`; from another repository or URL context, `.github/docs/automation-roles.md` means that same file in the organization `.github` repository. It owns operational health, sync, scheduler, CI/check, source-order, branch/default-branch drift, fleet refresh, and responsibility-alignment checks. Do not use this prompt as a general repo-improvement scout or readiness-review issue creator.

Use the local checkouts or worktrees provided by the Codex automation runtime instead of assuming a workstation-specific absolute path. Prefer running this monitor in a Codex worktree execution environment. The expected local repository names are `.github`, `builder-agent`, `kaizen-loop`, and `verifier`. If a concrete path is needed, derive it from the current checkout, the automation cwd list, or the parent directory of the `.github` checkout. If a local checkout cannot be located, report that local observation as unavailable and continue with GitHub remote checks.

Inspect local git status, local branch/upstream alignment, default-branch alignment, open GitHub PRs/issues, CI/check status where available, documentation/implementation alignment across the components, daily dogfood/shared-skill sync health, and local fleet refresh readiness. Treat local checkout state as observation only. Local files, local branches, dirty worktrees, and non-default remote branches may be useful evidence of drift, but they are not authoritative documentation basis for creating implementation issues. Do not make the provided local checkouts authoritative by checking them out, resetting them, cleaning them, or otherwise modifying user-visible branches.

Use the GitHub default branch, expected to be `origin/main` for these repositories, as the source of truth for documentation-backed findings. Before relying on any document as issue basis, fetch the target repository with `git -C <repo> fetch --prune origin` when a local checkout is available, then verify the cited file from the updated default-branch ref. Prefer reading and running default-branch checks from an isolated temporary worktree created from `origin/main`, for example `git -C <repo> worktree add --detach <tmp>/kaizen-monitor/<repo> origin/main`, and remove that worktree before finishing. If temporary worktree creation is unavailable, use `git -C <repo> show origin/main:<path>` or GitHub default-branch content instead. Do not use a local-only file, a file that exists only on a feature branch, or unmerged local content as documentation basis for an implementation issue. If local docs differ from `origin/main`, report that as local/default-branch drift. If a required source is missing from `origin/main`, stale, contradictory, or does not support the proposed scope, do not create an implementation issue from inference; include the drift or clarification need in the report instead, or create a documentation clarification issue only when that is the clear actionable next step and not already covered by an open issue or PR.

Path convention: when reading from the `kaizen-agents-org/.github` repository checkout or its default-branch ref, organization docs are repository-relative paths under `docs/...` and the organization README is `README.md`. When referring to those same files from another repository or URL context, `.github/docs/...` means the `docs/...` directory in `kaizen-agents-org/.github`, and `.github/README.md` means that repository's README.

Project-local docs may add repository-specific detail, but must not silently override the organization-level responsibility model. If project docs and organization docs disagree, report the drift instead of filing an implementation issue based on the conflict.

Evaluate every active monitored repository as a possible target, not only the repositories with the largest cross-repo symptom. For each repository, apply its documented responsibility model: `.github` owns organization documentation, automations, and sync source docs; `kaizen-loop` owns orchestration, workspaces, policy, verification command execution, PR creation, scheduling, and run metrics; `builder-agent` owns requirement understanding, implementation artifacts, self-review quality, adapter/CLI behavior, backend/model selection, fallback behavior, build-result schema fidelity, and outputs consumed by `kaizen-loop` and `verifier`; `verifier` owns independent review depth and verdict quality. Search existing open issues and PRs across all monitored repositories before creating any issue, not just in the proposed target repository. Produce a concise coordination report with: current repo health, verified GitHub issue/PR counts, existing monitor work, open PRs needing attention, cross-repo inconsistencies or drift, sync workflow health, fleet refresh readiness, repo-local candidate coverage, recommended next actions, and any safe follow-up work that should be done in a PR. For closed-loop health, check whether `.github` main changes to managed dogfood paths trigger `Sync Daily Dogfood Contracts`, whether shared-skill main changes trigger `Sync Kaizen Shared Skills`, whether target sync PRs are ready for review and linked to source issues when source issues exist, and whether local runner state can be refreshed from synchronized default-branch workspaces with `kaizen fleet --root .. --owner kaizen-agents-org --repo .github --repo builder-agent --repo kaizen-loop --repo verifier --prune --verify`. Existing open issues or PRs about branch-only work, missing documentation sources, sync drift, or monitor findings should normally prevent creating another issue for the same finding.
