# 実プロダクト投入プラン — 「いつから Kaizen Agents で外部プロダクトを改善し始めるか」(2026-07-05)

対象読者: このハーネスを実プロダクトに適用したいオーナー、および投入判断・準備作業を支援する AI エージェント。
根拠: [evaluation-2026-07-05-v2.ja.md](./evaluation-2026-07-05-v2.ja.md)(現状評価 A-)、[improvement-playbook.ja.md](./improvement-playbook.ja.md)(Phase A/B/C)、2026-07-05 のオーナーとの検討。
最終目的の確認(変更なし): dogfooding は手段、**外部プロダクトへのハーネス運用がゴール**。

---

## 1. 結論

**「完成を待ってから使う」のではなく、約 2 週間後に監視付きパイロットとして 1 プロダクトを載せ始めるのが最適。** 本格的な運用レベル(レビュー負荷が軽い定常運用)への到達は **1.5〜2.5 ヶ月後** が現実的な見立て。

改修と実運用投入は直列ではなく**並行**にすべきであり、その並行こそがプレイブック Phase B の設計そのものである(B-4「似ていないリポジトリでの dogfooding」= 実プロダクト投入で崩れる前提の洗い出し)。

## 2. なぜ「待つ」必要がないか

システムの安全設計上、失敗のコストが最初から抑えられているため:

- **自律マージは凍結中**(human-in-the-loop 必須)、PR-only モード、branch protection + conversation resolution 必須。最悪ケースは「main が壊れる」ではなく「レビュー時間が無駄になる」。
- **verifier が独立ゲート**として機能しており、誤検知(verifier #86 のような false positive)はあっても誤マージ方向の事故は起きにくい。
- **WIP 制限が実装済み**(`safety.wipLimit`)なので、パイロットで PR が積み上がっても intake が自動停止する。

さらに、実プロダクトを載せること自体が Phase B-4 の実施であり、待っていても得られない学習(スタック前提の崩れの発見)が載せることで得られる。

## 3. 段階別タイムライン(2026-07-05 起点、現在の velocity 前提)

| 時期 | 状態 | できること |
| --- | --- | --- |
| 今すぐ | Phase A ほぼ完了 | 技術的には載せられるが、メトリクス計測(kaizen-loop#158)がなく効果を測れない |
| **約 2 週間後(推奨開始点)** | 追跡 issue 群 + #158 消化、対象リポジトリで sandbox smoke 1 回完走 | **監視付きパイロット開始**。全生成 PR を人がレビュー、週次でメトリクス確認 |
| 1.5〜2.5 ヶ月後 | Phase B 完了(verifier LLM 化の実測改善、PR 証拠パッケージ kaizen-loop#172、非 Node 実績) | **定常運用レベル**。PR レビューが「5 分で判断できる」状態になり、レビュー帯域が制約でなくなる |
| 3 ヶ月〜 | Phase C(信頼境界の再設計 kaizen-loop#198/#200(旧 kaizen-loop#160/#174)、Actions reusable workflow kaizen-loop#199(旧 kaizen-loop#173)) | 外部コントリビュータの issue も受け付けられる、ほぼ手離れした運用 |

改善 velocity は高い(PR バックログ 12→1 件、WIP 制限実装、smoke 完走をそれぞれ 1 日で消化した実績)ため、Phase B の律速は実装量ではなく**「メトリクス実績の蓄積に必要な暦時間」と「オーナーのレビュー帯域」**である。

## 4. パイロット開始の前提条件(4 つ)

1. **kaizen-loop#158(メトリクス永続化)完了** — 効果測定なしのパイロットは学習にならない。North-star は人手修正なしマージ率。
2. **対象リポジトリで `kaizen smoke` を 1 回完走** — スタック前提(検証コマンド、ログパターン)の崩れを事前に洗い出す(手順: kaizen-loop `docs/13-sandbox-smoke.md`)。
3. **issue の起票者を信頼できる人に限定** — 信頼境界の再設計(kaizen-loop#198、旧 kaizen-loop#160)が終わるまで、Issue 本文は builder への実質的な指示として流れる。**外部からの issue 受付は Phase C-2 完了まで待つこと(これだけは待つべき項目)。**
4. **WIP 制限をパイロット用に低めに設定**(例: 2〜3)— オーナーのレビュー帯域に合わせる。

## 5. プロダクトの性質による補正

- **Node/TS のプロダクト**: 前提がほぼそのまま通用するため前倒し可能(2 週間 → 1 週間でも)。
- **Python/Rust/Go 等**: verifier の golden ケースに実ログが入り始めているとはいえ、検証コマンド推定(verifier#70)などの前提崩れが出るはず。それ自体が Phase B-4 の成果になるので載せる価値はあるが、**最初の 2〜3 件の Issue→PR は「動作確認」と割り切る**。
- **本番顧客のいるクリティカルなプロダクト**: マージ後 revert 率のデータが数週間分貯まるまで、リスクの低い issue(ドキュメント、テスト追加、小さなリファクタ)に限定して流す。

## 6. 評価の現在地と到達見込み

- 現在: **A-(設計 A / 運用の証明 B+)**([evaluation-2026-07-05-v2.ja.md](./evaluation-2026-07-05-v2.ja.md))
- 追跡 issue 群(.github#106/#107/#108、kaizen-loop#175、verifier#88/#89)+ kaizen-loop#158 + worktree 化(kaizen-loop#87/#88、.github#104)完了時: **A(設計 A / 運用の証明 A-)** — Phase A が完全クローズし、「人の努力ではなく仕組みで維持される」状態。
- A+ に必要なのは実装ではなく**運用実績の証明**: 週次メトリクスの蓄積(数週間分)、Phase B の verifier 意味判定で誤 block 率低下の実測、非 Node リポジトリ完走(.github#102)。

## 7. 投入手順の概要(パイロット開始時)

1. 対象リポジトリに `.kaizen/config.yml` を配置する。dogfood-sync targets は構造と検証コマンドの参考に限定し、組織外・第三者導入では `safety.operationMode: external` を明示して `operationMode: dogfood` を複製しない。`policy.mode` は PR-only、`safety.wipLimit` は低め。
2. branch protection を組織標準に揃える(required check + `required_conversation_resolution: true`。運用方針は .github#106 で文書化予定)。
3. `kaizen smoke` を完走させ、artifact を記録(§4-2)。
4. リスクの低い実 issue(ドキュメント/テスト/小リファクタ)から 2〜3 件流し、PR 品質と検証ログを確認。
5. 週次でメトリクス(人手修正なしマージ率 / time-to-merge / verifier block 率 / revert 率)をレビューし、改善傾向が見えたら流量を上げる。

## 8. まとめ

推奨は段階投入: **追跡 issue 群 + #158 を消化(〜2 週間)→ 対象プロダクトで smoke 完走 → 信頼できる起票者・低 WIP 制限・全件レビューの条件で開始 → メトリクスが人手修正なしマージ率の改善を示したら流量を上げる。** 唯一「待つべき」なのは、信頼できない起票者からの issue 受付(Phase C-2 完了まで)のみ。
