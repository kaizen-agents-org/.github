## 4. 発見事項(優先度順)

### 4.1 【最重要・回帰】builder の実行基盤が停止 — 全コマンドが起動前に失敗

07-10 以降、同一の根本原因(`~/.local/bin/codex-code-mode-host` バイナリの欠落: `No such file or directory (os error 2)`)で builder-agent に **5 件の issue(#115/#116/#118/#119/#120)** が別個に起票された。5 件の Evidence は同一文字列であり、実体は 1 つの環境障害である。加えて kaizen-loop#216(P1: 「sandbox が node/pnpm/tsx を全面ブロックし、無人実行に approver がいない」)と #217 も同系統。つまり:

- **修理は 1 箇所**(code-mode host の復旧 + sandbox 承認ポリシー)で済む可能性が高いが、
- **intake の重複排除が効いていない**ことが同時に露呈した。scout/monitor は同一症状の連続起票を検出できていない。これは外部運用時に issue ノイズとして顧客に直撃する種類の欠陥。

07-08 レポート G3 の「統制された自環境ですらこの状態なら、外部環境ではさらに脆い」という警告が、3 日で実データにより裏付けられた形。

### 4.2 【工程】07-08 レポートの G4 根拠は執筆時点で既に陳腐化していた

「`prompt.ts:24,95` が issue 本文を生のまま補間」という 07-08 の記述は、07-05 マージの PR #176(issue #160)で既に修正済みだった。ギャップ分析が origin/main の最新に対して検証されていなかったことを意味する。教訓は 2 つ: (1) **kaizen-loop#198 は起票時点で大半が済んでいた issue** — runner.ts:355 とエスケープ検証に再スコープするか close する。(2) 評価・準備レポートは公開前に主要な根拠行を HEAD で再確認する工程を挟む(本レポートは全根拠を 07-11 の origin/main / GitHub API で採取した)。

### 4.3 【メトリクス】北極星 100% は n=1 — 分母を常に併記する

W28 の「人手修正なしマージ率 100%」は検査対象の生成 PR が 1 件しかない。この数字を単独で引用すると自律マージ判断に誤った自信を与える。W28 の実質的な情報はむしろ **issue→PR 成功率 26%(19/73)と verifier ブロック率 51.3%(20/39)** で、「入口の半分が verifier で止まる」現状を初めて定量化した点にある。以後すべての率に分母を併記する規律を(#203 と合わせて)固定すべき。

### 4.4 【前回指摘の帰結】worktree 運用は定着、しかし ledger が止まった

07-05 v2 の最優先指摘(worktree 隔離)は `_pr-work/` 配下 36 worktree として採用され、機能している。一方 primary checkout の drift(kaizen-loop 3 behind、builder-agent 5 behind)は残る — worktree 運用下では実害が小さいため優先度は下げてよい。より気になるのは **ledger/current.md が 07-06 から更新されていない**こと。07-08〜07-11 に 15 本以上の PR がマージされたのに台帳に痕跡がなく、「進行中の目標と証拠を台帳に残す」という運用契約が途切れている。

### 4.5 【運用】cross-repo dispatch の権限ギャップ(kaizen-loop#208)

orchestrator が「他リポジトリへの Issue→PR→マージ」を要する issue を、単一リポジトリ権限しかない builder セッションに配ってしまう。外部適用時には「ハーネスがどこまでの操作を許可されているか」の境界そのものなので、C-2(#200)の一部として扱う価値がある。
