Kaizen Agents Org / Onboarding Flow

ハーネスを他プロジェクトへ導入するフロー

対象リポジトリで onboard.sh を 1 回実行すると、Onboarding Contract(導入完了の状態定義)が満たされ、以後は同じコマンドの再実行がアップデートになる — という設計。所要 30 分以内、前提は「gh CLI が使える」こと。

現在のステータス: 設計段階。このフローは PR #111 の設計書で定義されたもので、実装は Stage 1 の issue(.github #112 / #113、kaizen-loop #178 / #179)の完了待ちです。

展開の 3 段階

STAGE 1

Local Scheduler 型

オーナーのマシンで scheduler を回す。dogfood で稼働中の構成に「導入動線」を足すだけ。

実装対象(〜2週間)
STAGE 2

GitHub Actions 型

対象リポジトリの Actions で完結。caller workflow 1 枚 + kaizen:run ラベルが実行認可。

Phase B 完了後(C-1)
STAGE 3

公開テンプレート

「Use this template」で第三者がドキュメントのみで導入。

Phase C

入口 — 対象リポジトリで 1 コマンド

$ cd my-product
$ curl -fsSL https://raw.githubusercontent.com/kaizen-agents-org/.github/main/onboarding/onboard.sh | sh

兄弟スクリプトと versions.json は onboard.sh 自身と同じ base URL・同一 ref から取得。対象リポジトリ側には何も前提としない。冪等なので、済んでいるステップは検知してスキップされる。

onboard.sh の内部 — 7 ステップ・人間の確認は 3 点だけ

1

ツールチェーン install / update

versions.json(互換バージョン組)と突合し、kaizen-loop / builder-agent / verifier のうち差分があるものだけ更新。

install-kaizen.shversions.json
2

スタック検出 → プロファイル提案

package.json / pyproject.toml / go.mod / Cargo.toml を検出し、pilot-node / pilot-python / pilot-go / pilot-rust を提案。検出は提案まで — 確定はしない。

確認 1プロファイルと verify コマンドの承認(検証コマンドは信頼の根幹)
3

kaizen init --profile <選択>

プロファイルをデフォルト config にオーバーレイして雛形を生成。ラベル・issue テンプレート作成、shared skills の vendor と指紋記録まで行う。

.kaizen/config.ymlkaizen ラベル群ISSUE_TEMPLATE/kaizen.ymlskills/ + manifest
4

Branch protection 適用

組織標準(required check + conversation resolution + enforce_admins)を gh api PUT 1 回で適用する冪等スクリプト。

確認 2管理権限に踏み込む操作のため明示承認
5

kaizen doctor --repair

プリフライト: gh 認証・ラベル・agent 認証・builder / verifier の可用性を検査し、直せるものは自動修復。

6

kaizen smoke --yes

sandbox の Issue → PR を実際に完走させる受け入れテスト。完走の証拠を artifact としてリポジトリにコミット。

確認 3実リポジトリに sandbox issue / PR を作る操作のため承認
docs/smoke-runs/<ts>.json
7

check-onboarding-contract.sh

契約充足を決定的スクリプトで最終判定。組織の安全最低線 — pr-only 固定、wipLimit ≤ 5、verifier 有効、protectedPaths がオーバーレイで削れない最低集合を包含 — もここで検証する。

✓

導入完了 = Onboarding Contract 充足

手順書を終えたからではなく、対象リポジトリが次の状態になったことをもって完了と機械判定する:

  • .kaizen/config.yml(スキーマ + 安全最低線)
  • kaizen ラベル群
  • issue テンプレート
  • branch protection
  • skills + skills-manifest.json
  • smoke artifact 1 件以上

導入後の運用 — 同じ契約の上にランタイムを載せ替える

形態 A — STAGE 1(今すぐ)

Local Scheduler 型

オーナーのマシンで夜間などに scheduler を実行。初期は流量を絞る(wipLimit 2、1 夜 1 issue)。

kaizen scheduler
└→ kaizen fix <issue>
   ├ builder-agent
   ├ 機械検証(config)
   └ verifier → PR 作成
形態 B — STAGE 2(Phase C-1)

GitHub Actions 型

書き込み権限者が issue に kaizen:run ラベルを付ける = 実行認可。対象側の導入物は caller workflow 1 枚。

issue labeled: kaizen:run
└→ kaizen-run.yml(caller)
   └→ reusable workflow @v0.x
      contract check → fix → PR
どちらの形態も同じ Onboarding Contract・同じ PR 品質ゲート(機械検証 + verifier + 人間レビュー)に合流する。自律マージは凍結のまま。

継続運用

↻

アップデート = 同じコマンドの再実行

versions.json 突合 → 差分コンポーネントのみ更新 → skills 再 vendor → config マイグレーション(ユーザーのカスタマイズは保持)→ 契約再検証。専用のアップデート手順は存在しない。

◔

週 1 の self-update check

upstream に新しい互換バージョン組が出たら対象リポジトリに通知 issue を 1 件だけ立てる。自動更新はせず、実行は常にオーナーの 1 コマンド。

＋

fleet.json への登録(1 行 PR)

org 側の monitor / weekly readiness レビューの巡回対象に加わる。automation のコピーは発生しない。scout(issue 生成側)は消化の安定を確認してから opt-in。
