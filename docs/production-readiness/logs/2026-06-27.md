# Production Readiness Review: 2026-06-27

## Scope

Reviewed the local Kaizen Agents repository set:

- `kaizen-agents-org/.github`
- `kaizen-agents-org/kaizen-loop`
- `kaizen-agents-org/builder-agent`
- `kaizen-agents-org/verifier`
- `kaizen-agents-org/coderabbit`
- `kaizen-agents-org/renovate-config`

The checkout root was a collection of repositories, not a single git repository. Existing local worktree state was treated as observation only and was not reverted.

## Summary

Kaizen Agents is a strong MVP for the issue-to-ready-for-review-PR loop, but it
is not yet production-ready automation. The main blockers are verifier depth,
repeated real end-to-end evidence, safety hardening, and operational
observability.

## Findings

1. Verifier is still an MVP gate, not a production-quality independent reviewer.

   Evidence: `verifier/README.md` describes the current implementation as an
   MVP gate. The implemented verdict logic in
   `packages/core/src/minimal-verdict.ts` is mostly deterministic pattern
   scanning over task text, diff text, verification logs, and builder reports.
   The richer staged verifier described in `verifier/docs/SPEC.md` and
   `verifier/docs/DESIGN.md` is still roadmap work.

   Impact: The system can conservatively decide whether to open a reviewable PR, but it cannot yet claim high-confidence merge-readiness assessment.

   Needed next step: Build the first executable verifier eval harness and
   seeded-bug corpus, then expand the verifier toward intent claims, evidence
   collection, and false-positive measurement.

2. Repeated real end-to-end smoke evidence is missing.

   Evidence: `kaizen-loop` has useful mocked integration coverage for the PR
   flow, and organization docs already identify repeated end-to-end smoke tests
   as remaining hardening work. The observed tests do not prove a real GitHub
   sandbox issue can repeatedly move through builder-agent, mechanical
   verification, verifier, PR creation, issue-link recognition, and PR guardian
   follow-up.

   Impact: The highest-risk integration boundaries are still external: `gh`, actual providers, branch pushing, GitHub issue linkage, and PR follow-up.

   Needed next step: Add a controlled sandbox repository smoke test and record each run's issue, branch, PR, verification output, verifier verdict, and linkage check.

3. Safety design and implementation are not fully aligned.

   Evidence: Safety docs require process-tree termination, minimal environment
   exposure to agents, disk-space preflight, and graceful shutdown behavior.
   Current command execution kills the immediate child process, agent and shell
   commands inherit broad `process.env`, and no disk-space preflight
   implementation was observed.

   Impact: Unattended runs can leave child processes alive, expose more environment than intended, or fail late under resource pressure.

   Needed next step: Implement process-group termination, environment allowlisting, run-level timeout enforcement, disk-space checks, and shutdown cleanup tests.

4. Generated PR readiness checks are not strict enough yet.

   Evidence: `GitHubClient.createPullRequest` creates the PR and parses the URL.
   The organization MVP contract also says generated PRs should be checked for
   default-branch targeting and GitHub-recognized `closingIssuesReferences`
   before the PR is considered ready.

   Impact: A PR can appear to be created successfully while failing to close its source issue on merge, or while targeting the wrong base branch.

   Needed next step: After PR creation, verify `baseRefName`, `isDraft`, and `closingIssuesReferences` with bounded retries, mirroring the stricter sync workflow behavior.

5. Operational metrics are still too thin for sustained use.

   Evidence: Status, logs, summaries, and monitor docs exist, but
   production-readiness indicators such as PR success rate, verifier block rate,
   needs-human rate, average processing time, repeated failure reasons, and
   post-merge correction or revert rate are not yet first-class release gates.

   Impact: The system can run, but it is hard to know whether quality is improving or degrading over time.

   Needed next step: Add metrics that separate selection, builder, verification, verifier, PR creation, guardian, and human-review outcomes.

## Priority

Recommended order:

1. Make `verifier` local verification reproducible.
2. Add real sandbox E2E smoke runs and keep dated results.
3. Harden safety controls that affect unattended execution.
4. Enforce post-PR issue-link and base-branch checks.
5. Add operational metrics and use them in readiness reviews.

## Current Readiness Judgment

Ready for continued dogfooding and review-required PR generation.

Not ready for claims of production-grade autonomous maintenance, high-confidence independent verification, or unattended direct-commit operation.
