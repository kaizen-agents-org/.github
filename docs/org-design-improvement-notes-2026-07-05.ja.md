## 2. 組織設計そのものの改善点(実装 TODO ではなく設計論点)

以下は個別リポジトリのバグや未実装機能ではなく、**複数コンポーネントに共通する設計上のギャップ**として指摘する。いずれも「新しい仕組みを追加せよ」ではなく、**既にどこか一つのコンポーネントで実現できている良い設計判断を、組織全体の原則として横展開する**という方向性。

### 2.1 スループット制御が「生成側」にしか存在しない

scout(1日3回)・monitor・readiness-issue-creator は issue を作り続けるが、消化速度(人間のレビュー帯域)と突き合わせるフィードバックループが設計上存在しない。生成と消化が非同期な片方向パイプラインになっており、構造的にバックログが積み上がりやすい。

**設計原則として欠けているもの**: 「open PR 数が上限を超えたら生成側を自動的に止める」というプル型の負荷制御弁。トヨタ生産方式のかんばん(WIP 制限)に相当する仕組みが、生成側と消化側を跨いで存在しない。IMPROVEMENT-PLAYBOOK.ja.md A-3 で既に認識済みだが未実装。

### 2.2 「誰が何を検証したか」の証拠の階層が未分化

verifier に導入された `evidence_grade`(executed/reported)は正しい方向だが、これは verifier 戦略ノートが定義した証拠強度階層(runtime > test > static > reading)の一部に過ぎない。現状、builder の自己申告・機械検証ログ・verifier 判定が **PR 本文上では同じ強度の情報として並んでしまう**。人間レビュアーが「実際に確認された事実」と「自己申告」を一目で区別できる表示になっていない限り、レビュー時間短縮(価値式の分母)は改善しない。

**設計原則として欠けているもの**: PR 本文テンプレートに、各主張の証拠強度ラベル(実行済み/報告のみ/未検証)を必須項目として組み込む一貫した表示規約。

### 2.3 「意図の一次ソース優先」原則が verifier にしか実装されていない

verifier の SPEC は「Issue 本文を一次ソース、PR 説明・コミットメッセージを二次ソースとし、二次ソースのみでは mergeable に到達させない」という優れた設計判断を持つ。しかしこれは verifier 内部の判定ロジックにしか実装されておらず、**kaizen-loop が builder にプロンプトを渡す段階では同じ区別がされていない**可能性が高い(builder は Issue 本文を「指示」として受け取っており、「参照データ」として明示的に分離されているかは未確認)。

**設計原則として欠けているもの**: 「Issue 本文は指示ではなくデータである」という区別を、verifier だけでなくパイプライン全体(特に intake → builder プロンプト生成)に一貫して適用する規約。これは外部ハーネス運用時の prompt injection 対策としても直結する(改善プレイブック Phase C-2 の前倒しの根拠になる)。

### 2.4 自動化コンポーネント間の役割重複

`.github` 配下に scout・monitor・readiness-review・readiness-issue-creator の 4 自動化が並存し、`docs/automation-roles.md` で責務分担は文書化されているものの、**出力(issue 作成)という観点ではすべて同じ末端動作**を持つ。同じ問題が複数の自動化経由で重複 issue 化されるリスクが構造的に存在する。

**設計原則として欠けているもの**: 「1つの問題は1つの自動化だけが issue 化する」という排他制御(既存 open issue とのタイトル/内容類似度チェックなど)。scout の頻度を下げる運用対応(プレイブック A-3-1)は対症療法であり、根本には重複検出の仕組みが必要。

### 2.5 「判定はコードで、発見は LLM で」の原則が verifier にしか明文化されていない

verifier の DESIGN.md にある「Verdict 決定・confidence 算出は決定的ルールにし、LLM は Claim 抽出・Finding 発見・反証という発散的タスクのみ担当する」という原則は、実は kaizen-loop の他の判断ロジック(`policy.mode` の決定、`pr-guardian` の判断、`goal` runner の完了判定)にも同型で適用できるはずだが、明文化されているのは verifier だけ。

**設計原則として欠けているもの**: この原則をコンポーネント横断の設計指針として `.github/docs/design-decisions.md` に格上げすること。今後 kaizen-loop や builder-agent が新しい自動判断ロジックを追加する際、「その判断はコードで固定するのか、それとも LLM に委ねるのか」を最初に問う一貫したチェックリストになる。

## 3. 優先順位の示唆

上記 5 点はすべて「今すぐ壊れている」ものではなく、**今の急改善(07-04→07-05)が偶然ではなく持続可能になるための予防線**である。実装労力の小さい順に着手するなら:

1. **§2.5(design-decisions.md への原則格上げ)** — ドキュメント追記のみ、実装ゼロ。次に新しい判断ロジックを書く誰か(人間でも AI でも)への最も安いガードレール。
2. **§2.1(WIP 制限)** — プレイブック A-3 に既に実装方針あり。`config` に `wipLimit` フィールドを追加し、issue 選択時にチェックするだけの小さな変更。
3. **§2.4(自動化の重複検出)** — scout/monitor の issue 作成前に既存 open issue とのタイトル類似度チェックを追加。
4. **§2.2(証拠強度の表示規約)** — PR 本文テンプレートの拡張(改善プレイブック B-3 と統合可能)。
5. **§2.3(意図の一次ソース原則の横展開)** — kaizen-loop の intake/prompt 生成コードの変更を伴うため最も大きい。Phase C-2 (信頼境界の再設計)と合流させるのが効率的。
