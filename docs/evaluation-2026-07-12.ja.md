## 4. 発見事項(優先度順)

### 4.1 【最重要・停滞】B-4(非 Node 完走)が 4 レポート連続未着手

07-04 / 07-08 / 07-11 / 本レポートと、外部適用ゴールのクリティカルパスが一度も動いていない。今回状況が変わったのは、**verifier 側の準備が整った**ことだ(コーパスに pytest/cargo/go/gradle の非 Node ケース約 19 件が入り、B-1 完了)。残る前提は kaizen-loop#179(init の非 Node stack 検出)のみで、これは pyproject/go.mod/Cargo.toml/Gemfile の検出という限定的な実装。ループ自身が 1 日 9 PR を出せる状態に戻った今、#179 をループに食わせない理由がない。

### 4.2 【衛生】P1 issue #216 が現実から遅延 — 「直ったのに open」は「壊れたのに open」と同じくらい有害

kaizen-loop#216(P1: sandbox が node/pnpm/tsx を全面ブロック)と #217 は open のままだが、07-12 に 33 PR がマージされた事実と矛盾する。builder#122/#124 は host 欠落側(#217 系)への対処であり、#216 固有の主張(無人実行に approver がいない)がどの変更で解消されたのかを明示的に検証した記録がない。無人 run を 1 回流して sandbox 実行を確認し、close または実態に合わせて再スコープすべきだ。P1 が「実は直っている」状態で残ると、次のトリアージで優先度判断のノイズになる。

### 4.3 【工程】intake dedup が issue 化されていない — 手動対応で終わり、仕組み化が漏れた

07-11 の緊急アクション 1 の後半「同一 Evidence 文字列の連続起票を抑止する intake 側 dedup を issue 化する」が実施されていない。5 重複の統合 close は完了したが、これは症状の掃除であって再発防止ではない。外部運用時に顧客のリポジトリへ同一障害の issue を 5 連投する欠陥は未対策のまま残っている。数分で起票できる。

### 4.4 【文書】正本化契約(.github#107)が 4 日で破綻している

07-06 に「.github/docs/ を評価レポート・プレイブックの正本とする」と決定・close したが、実際には 07-08 以降の更新(07-11 評価レポート、playbook の 07-11〜12 進捗ログ、運用ルール 2 原則)はすべて org ルートの非バージョン管理コピーにのみ存在する。canonical 側の playbook は 07-08 で停止。「決定はしたが同期する仕組みがない」典型で、07-11 に playbook へ追加した 2 原則(検証 ref 明記 / expected を出力に合わせない)がバージョン管理に乗っていないのは皮肉な状態。daily dogfood sync の対象に playbook/評価レポートを含めるか、org ルート側を捨てて .github/docs 直編集に切り替えるか、どちらかに倒す。

### 4.5 【好例として記録】knownGap 機構と #198 の close は「評価の質」の改善

今サイクルで最も価値が高いのは、単発の修正ではなく工程の改善が 2 つ定着したこと。(1) eval の expected を実装のバグに合わせず、`knownGap` フラグ + 追跡 issue(verifier#104)で ground truth を保持する機構が EVAL.md に文書化された。(2) #198 close は「issue の文言通りに直す」のではなく、データフローを遡って G4 非該当を証明した。どちらも 07-08 レポートの stale citation 問題(検証なき主張)への構造的な回答になっている。
